# i18n Audit — 2026-04-07

## TL;DR

Psycologger has **zero** i18n infrastructure. No `next-intl`, no `react-i18next`, no message catalog. Every user-facing string is hardcoded inline, mostly in pt-BR but with sporadic English leakage. This is fine for a Brazil-only product today but will become expensive to fix the longer it's deferred.

## Findings

1. **No i18n library installed.** `grep -r "next-intl\|i18next\|useTranslation" src/` returns nothing. There's no `messages/` or `locales/` directory. Strings live where they're rendered.

2. **Locale is pt-BR by default** in most user-facing places — patient portal, dashboard widgets, form labels, error toasts, email templates (`src/lib/email.ts`). Currency is BRL, dates use `date-fns` with `pt-BR` locale, CPF formatting is Brazil-specific.

3. **English leakage spots** found in spot-check (non-exhaustive — a real audit needs a string-extraction tool):
   - Staff app navigation labels in `src/components/shell/app-sidebar.tsx` (mix of "Dashboard", "Patients" vs "Pacientes")
   - Empty-state copy in some patient list components
   - Several `Button` labels in admin areas
   - All Sentry breadcrumb messages and audit-log action names (these are technical, English is fine)
   - Error messages thrown from server — many are in English ("Unauthorized", "Not found", "Forbidden") but some in pt-BR. Inconsistency hurts users when an error toast surfaces them raw.

4. **Date/time formatting** is consistent — uses `date-fns-tz` with `America/Sao_Paulo` and `ptBR` locale. No issue here.

5. **Email templates** in `src/lib/email.ts` are pt-BR. Good.

## Cost of doing nothing

- Onboarding a non-Brazilian clinic = re-touching every component file. With ~100+ component files, this is a 1–2 week project, not an afternoon.
- Translation memory (so you don't re-translate the same string in multiple places) doesn't exist — fixes will be inconsistent.

## Recommendation

**Don't fix today.** The pragmatic call is:

1. Add a lint rule (eslint-plugin-i18next has a `no-literal-string` rule) configured to **warn** (not error) on user-facing JSX text. This puts visibility on new English strings being added.
2. When you actually need a second locale (likely 2026 H2 if at all), do it with `next-intl` because you're on App Router. Estimate: 1.5 weeks for one full locale on the current surface.
3. In the meantime, normalize all server-thrown error messages to pt-BR so when one bubbles up to a user toast, it doesn't mix languages. This is a 2–3 hour job (grep server routes for `throw new Error("English string")` and translate). Not done in this pass.

## Concrete English-leakage hits (sample, not full list)

Run this to see them yourself:

```bash
grep -rn '"[A-Z][a-z]\+ [a-z]\+\|>[A-Z][a-z]\+ [a-z]' src/app/app src/components --include='*.tsx' | grep -v 'pt-BR\|className\|aria-' | head -50
```

This is a heuristic — false positives include `className` strings, prop names, and code identifiers — but it surfaces the obvious offenders quickly.
